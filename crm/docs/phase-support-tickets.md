# ماژول پشتیبانی / تیکت‌ها (Support / Ticketing)

> وضعیت: **اجراشده و اعتبارسنجی‌شده.** بک‌اند و فرانت build سبز، ۹۰ تست واحد، ۱۷ سنجه‌ی
> API e2e، و ۱۱ سنجه‌ی مرورگر (Playwright) — همه سبز. اولین ماژول سازمانی پس از سخت‌سازی تولید.

## ۱) چه ساخته شد
- **بک‌اند `TicketsModule`** روی اسکیمای موجود `Ticket`/`TicketComment` (بدون بازنویسی):
  CRUD، گردش‌کار وضعیت با اعتبارسنجی انتقال، تخصیص مسئول، کامنت/یادداشت داخلی، آمار، و SLA.
- **دسته‌بندی هوشمند (AI)**: `ticket-classify.ts` — موتور قاعده‌محورِ فارسی که از موضوع/توضیحات
  **دسته** (خرابی/نصب/سرویس/گارانتی/شکایت/استعلام)، **اولویت** (با تشخیص خطر ایمنی مثل «بوی
  گاز» ⇒ فوری)، و **تجهیز HVAC مرتبط** (دیگ/مشعل/پمپ/…) را تشخیص می‌دهد. بدون نیاز به LLM؛
  با فعال‌کردن مدل واقعی (Ollama) قابل ارتقا.
- **ثبت بینش**: هر تیکت یک `AIInsight` از نوع `TICKET_CLASSIFICATION` می‌گیرد (مهاجرت 0005).
- **Realtime**: انتشار `ticket:created`/`ticket:updated` از `RealtimeGateway` به نقش‌های
  پشتیبانی و مسئول تیکت؛ با آداپتور Redis (H4) بین همه‌ی اینستنس‌ها پخش می‌شود.
- **ممیزی**: ساخت/تغییر وضعیت/تخصیص/حذف در `ActivityLog` ثبت می‌شود (H1).
- **فرانت فارسی/RTL**: صفحه‌ی فهرست (آمار، فیلتر وضعیت/دسته، جست‌وجو، «تیکت‌های من»،
  صفحه‌بندی)، فرم ساخت با جست‌وجوی مشتری، و صفحه‌ی جزئیات (گفت‌وگو، گردش‌کار وضعیت، تخصیص،
  و پنل «دسته‌بندی هوشمند»). اشتراک realtime برای به‌روزرسانی زنده بدون polling.

## ۲) نقاط پایانی API (همه تحت RBAC)
| متد | مسیر | مجوز | کارکرد |
|---|---|---|---|
| POST | `/api/tickets` | `tickets:write` | ساخت (دسته/اولویت خودکار در صورت نبود) |
| GET | `/api/tickets` | `tickets:read` | فهرست: صفحه‌بندی + فیلتر (status/priority/category/customer/assignee/mine/open) + `q` |
| GET | `/api/tickets/stats` | `tickets:read` | آمار خلاصه (byStatus + overdue) |
| GET | `/api/tickets/:id` | `tickets:read` | جزئیات با کامنت‌ها |
| PATCH | `/api/tickets/:id` | `tickets:write` | ویرایش فیلدها |
| PATCH | `/api/tickets/:id/status` | `tickets:write` | تغییر وضعیت (با اعتبارسنجی انتقال) |
| PATCH | `/api/tickets/:id/assign` | `tickets:write` | تخصیص/لغو تخصیص مسئول |
| POST | `/api/tickets/:id/comments` | `tickets:write` | افزودن پاسخ/یادداشت |
| DELETE | `/api/tickets/:id` | `tickets:delete` | حذف |

## ۳) گردش‌کار وضعیت (state machine)
```
OPEN        → IN_PROGRESS | WAITING | RESOLVED | CLOSED
IN_PROGRESS → WAITING | RESOLVED | CLOSED
WAITING     → IN_PROGRESS | RESOLVED | CLOSED
RESOLVED    → CLOSED | IN_PROGRESS (بازگشایی)
CLOSED      → IN_PROGRESS (بازگشایی)
```
`RESOLVED` ⇒ `resolvedAt` ثبت می‌شود؛ `CLOSED` ⇒ `closedAt`؛ بازگشایی به `IN_PROGRESS` هر دو را پاک می‌کند.

## ۴) تغییرات دیتابیس
- **مهاجرت `0005_ticket_insight_type`**: افزودن مقدار `TICKET_CLASSIFICATION` به enum `InsightType`
  (افزودنی، `ADD VALUE IF NOT EXISTS` — سازگار با عقب).
- **بدون تغییر جدول**: مدل‌های `Ticket`/`TicketComment` از فاز ۲ بدون تغییر استفاده شدند.
- **seed**: مجوز `tickets:delete` افزوده شد (به‌صورت idempotent؛ مدیر از طریق ALL می‌گیرد).

## ۵) دسته‌بندی هوشمند — منطق
- **دسته**: بیشترین تطابق کلیدواژه‌ای بین شش دسته؛ پیش‌فرض امن `INQUIRY` با اطمینان صفر.
- **اولویت**: خطر ایمنی (بوی گاز/آتش/…) ⇒ `URGENT`؛ خرابی/بدون‌حرارت ⇒ `HIGH`؛ استعلام ⇒ `LOW`؛
  در غیر این‌صورت `MEDIUM`.
- **تجهیز**: نگاشت کلیدواژه به `ProductCategory` (هم‌راستا با موتور پیشنهاد محصول تماس‌ها).
- کاربر می‌تواند دسته/اولویت را صریح تعیین کند؛ در این‌صورت انتخاب کاربر حفظ می‌شود و بینش AI
  همچنان برای شفافیت ثبت می‌شود.

## ۶) شواهد اعتبارسنجی
- **واحد**: `npx jest` ⇒ **۹۰/۹۰** سبز (شامل ۸ تست دسته‌بند + ۱۱ تست سرویس تیکت).
- **API e2e (زنده)**: **۱۷/۱۷** — دسته‌بندی خودکار (خرابی/زیاد؛ بوی گاز⇒فوری)، ثبت بینش،
  فیلتر، انتقال مجاز/نامجاز وضعیت، تخصیص، کامنت، آمار، اعتبارسنجی، و حذف.
- **مرورگر (Playwright)**: **۱۰/۱۰** — ساخت با دسته‌بندی خودکار، پنل هوشمند، تغییر وضعیت، ثبت
  پاسخ، واگذاری؛ به‌علاوه **۱ سنجه‌ی realtime**: تیکتِ ساخته‌شده از بیرونِ مرورگر بدون رفرش و
  به‌صورت push در فهرست ظاهر شد.
- اسکرین‌شات صفحه‌ی جزئیات بازبینی شد (RTL، برند SHOFAZH، نشان‌ها، پنل دسته‌بندی هوشمند).

## ۷) سازگاری عقب‌رو و Impact
- کاملاً افزودنی: ماژول جدید + یک مقدار enum + یک مجوز seed + یک متد به gateway. هیچ ماژول
  موجودی بازنویسی نشد و هیچ قرارداد API موجودی تغییر نکرد.
- افزودن `TicketsModule` به `AppModule`؛ `RealtimeProvider` فرانت رویدادهای تیکت را هم گوش می‌دهد.
- placeholder قبلی صفحه‌ی «تیکت‌ها» با صفحات واقعی جایگزین شد (سایر آیتم‌های منو دست‌نخورده).
