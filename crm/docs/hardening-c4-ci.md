# سخت‌سازی C4 — خط‌لوله‌ی CI

> وضعیت: **اجراشده و اعتبارسنجی‌شده محلی** (همه‌ی گام‌ها محلی سبز؛ YAML معتبر).
> اولویت: 🔴 Critical. فایل: `.github/workflows/crm-ci.yml`.

## ۱) مسئله
هیچ CI خودکاری برای CRM وجود نداشت؛ گردش‌کارهای موجودِ repo فقط مربوط به SEO/WordPress بودند.
هر تغییر بدون شبکه‌ی ایمنی خودکار وارد می‌شد.

## ۲) راهکار — چهار job مستقل (فقط روی تغییر `crm/**`)
| Job | گام‌ها | چه چیزی را تضمین می‌کند |
|---|---|---|
| **api** | `npm ci` → `prisma generate` → `prisma validate` → `build` → `test --ci` | کامپایل بک‌اند + صحت اسکیمای Prisma + ۶۲ تست واحد |
| **migrations** | راه‌اندازی PostgreSQL ۱۶ سرویس → `migrate deploy` → `migrate status` | مهاجرت‌ها روی DB واقعی تمیز اعمال می‌شوند و drift ندارند |
| **web** | `npm ci` → `typecheck` (tsc) → `build` (Next.js standalone) | صحت نوع‌ها و ساخت فرانت |
| **docker** | `docker build` برای api و web | درستی هر دو Dockerfile (build بدون push) |

ویژگی‌ها: `concurrency` با لغو اجراهای قدیمی، کش npm به‌ازای هر اپ، `CHECKPOINT_DISABLE`
برای خاموشی تله‌متری Prisma، اجرا روی `push` همه‌ی شاخه‌ها و `pull_request`.

## ۳) شواهد اعتبارسنجی (محلی، پیش از کامیت)
- `prisma validate` (با `DATABASE_URL` ست‌شده) ⇒ «schema is valid 🚀».
- `npm run build` (api) ⇒ بدون خطا؛ `npm test -- --ci` ⇒ **۶۲/۶۲** سبز.
- `migrate deploy` روی PostgreSQL محلی ⇒ هر سه مهاجرت اعمال شد.
- `typecheck` (web) ⇒ بدون خطا؛ `npm run build` (web) ⇒ موفق.
- YAML گردش‌کار با parser بررسی و معتبر شد.

## ۴) نکته‌ی لینت (تصمیم آگاهانه)
اسکریپت `lint` در هر دو اپ به ESLint اشاره دارد، اما **ESLint نصب/پیکربندی نشده** بود
(نه وابستگی، نه فایل پیکربندی) و هرگز عملیاتی نبود. برای سبز و صادقانه‌ماندن CI، گام لینت
فعلاً حذف شد و **«راه‌اندازی ESLint (flat config) + رفع تخلف‌ها»** به‌عنوان آیتم کیفیت کد
(Medium) در نقشه‌راه ثبت شد تا جداگانه و کامل انجام شود.

## ۵) سازگاری عقب‌رو
کاملاً افزودنی؛ فقط فایل گردش‌کار جدید. هیچ کد اپ تغییر نکرد. گردش‌کارهای SEO موجود
به‌دلیل فیلتر `paths: crm/**` متأثر نمی‌شوند.
