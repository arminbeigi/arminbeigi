# سخت‌سازی H6 — فشرده‌سازی پاسخ + تنظیم استخر/مهلت‌های Prisma

> وضعیت: **اجراشده و اعتبارسنجی‌شده** (کاهش ۷۸٪ حجم پاسخ زنده + ۷۱ تست سبز). اولویت: 🟠 High.

## ۱) چه چیزهایی اضافه شد
1. **فشرده‌سازی پاسخ (`compression`)**: میدلور gzip/deflate برای پاسخ‌های بالای آستانه‌ی ۱KB.
   پهنای‌باند پاسخ‌های JSON بزرگ را به‌شدت کاهش می‌دهد.
2. **تنظیم استخر اتصال Prisma**: پارامترهای اتصال در `DATABASE_URL` (در compose):
   - `connection_limit=${DB_POOL_SIZE:-10}` — اندازه‌ی استخر به‌ازای هر اینستنس API.
   - `pool_timeout=20` — حداکثر ثانیه انتظار برای اتصال آزاد پیش از خطا.
   - `connect_timeout=10` — مهلت اتصال اولیه (fail-fast اگر DB در دسترس نباشد).
   - متغیر `DB_POOL_SIZE` در `.env.example` با راهنمای ظرفیت
     (`DB_POOL_SIZE × تعداد اینستنس < max_connections پستگرس`).

## ۲) فایل‌های تغییر
- `src/main.ts` — افزودن `app.use(compression())` (پیش از parserهای بدنه).
- `docker-compose.yml` — پارامترهای استخر/مهلت در `DATABASE_URL`.
- `.env.example` — متغیر `DB_POOL_SIZE` با توضیح.
- `package.json` — افزودن `compression` (+ `@types/compression`).

## ۳) شواهد اعتبارسنجی
- `npm run build` ⇒ بدون خطا. `npx jest` ⇒ **۷۱/۷۱** سبز. رگرسیون e2e مشتریان ⇒ **۲۰/۲۰**.
- **زنده (فشرده‌سازی):** پاسخ ۲۶۵۱ بایتیِ `/api/audit` با `Accept-Encoding` روی **۵۸۸ بایت**
  روی سیم کاهش یافت (~۷۸٪)؛ هدر `Vary: Accept-Encoding` حاضر است.
- `docker compose config` ⇒ معتبر.

## ۴) سازگاری عقب‌رو
افزودنی و شفاف؛ کلاینت‌هایی که `Accept-Encoding` نمی‌فرستند پاسخ بدون فشرده‌سازی می‌گیرند.
پارامترهای استخر مقادیر محافظه‌کارانه‌ی پیش‌فرض دارند و قابل تنظیم‌اند.

## ۵) تنظیم آینده (اختیاری)
`statement_timeout` در سطح نقش/اتصال برای محدودکردن کوئری‌های فراری — آگاهانه به تعویق افتاد
تا با مهاجرت‌ها/پشتیبان‌گیری سنگین تداخل نکند؛ در صورت نیاز در سطح role قابل افزودن است.
