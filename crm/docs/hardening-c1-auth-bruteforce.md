# سخت‌سازی C1 — محافظت brute-force احراز هویت

> وضعیت: **اجراشده و اعتبارسنجی‌شده** (۱۰ تست واحد + ۲ اسکریپت e2e زنده سبز).
> اولویت: 🔴 Critical (نقشه‌راه آمادگی تولید).

## ۱) مسئله
مسیر عمومی `/api/auth/login` پیش‌تر هیچ محدودیت نرخ یا قفل حسابی نداشت؛ مهاجم می‌توانست با
حدس متوالی رمز (brute-force / credential-stuffing) حساب‌ها را هدف بگیرد.

## ۲) راهکار — دفاع دولایه
1. **محدودیت نرخ به‌ازای IP** (`@nestjs/throttler`):
   - سراسری: ۱۰۰ درخواست در دقیقه برای هر IP (`ThrottlerModule.forRoot` + `ThrottlerGuard` به‌عنوان `APP_GUARD`).
   - مسیرهای حساس (`login`/`register`/`refresh`/`logout`): سقف سخت‌گیرانه‌ی **۱۰ تلاش در دقیقه**
     با `@Throttle` در سطح هر متد. عبور از سقف ⇒ پاسخ **429 Too Many Requests**.
   - `app.set('trust proxy', 1)` در `main.ts` تا پشت Nginx، IP واقعی از `X-Forwarded-For`
     خوانده شود (هر کاربر سطل مستقل، نه یک سطل مشترک).
2. **قفل موقت حساب** (سطح هویت، مکمل لایه‌ی IP):
   - دو ستون جدید روی `User`: `failedLoginAttempts` و `lockedUntil` (مهاجرت `0003_auth_lockout`).
   - پس از **۵** تلاش ناموفق متوالی، حساب برای **۱۵ دقیقه** قفل می‌شود؛ در این بازه حتی رمز
     درست هم با پیام فارسی شفاف رد می‌شود.
   - ورود موفق، شمارنده و قفل را صفر/پاک می‌کند.

> چرا دولایه؟ throttler مبتنی بر IP، حملات توزیع‌شده روی یک حساب از چند IP را پوشش نمی‌دهد؛
> قفل حساب این شکاف را می‌بندد. در مقابل، قفل حساب به‌تنهایی مستعد سوءاستفاده‌ی DoS روی حساب
> قربانی است؛ ترکیب دو لایه تعادل مناسبی می‌دهد.

## ۳) فایل‌های تغییر/افزوده
- `prisma/schema.prisma` — دو ستون به `User` (افزودنی، سازگار با عقب).
- `prisma/migrations/0003_auth_lockout/migration.sql`.
- `src/app.module.ts` — `ThrottlerModule` + `ThrottlerGuard` (پیش از سایر گاردها).
- `src/modules/auth/auth.controller.ts` — `@Throttle` روی مسیرهای حساس.
- `src/modules/auth/auth.service.ts` — منطق قفل/شمارش در `validateCredentials`.
- `src/main.ts` — `trust proxy`.
- `src/modules/auth/auth.service.spec.ts` — ۵ تست جدید (شمارش، قفل، رد ورود قفل‌شده، صفرشدن).

## ۴) شواهد اعتبارسنجی
- `npm run build` ⇒ بدون خطا. `npx jest` ⇒ **۵۹/۵۹** سبز.
- **e2e زنده (throttle):** ۱۰ پاسخ ۴۰۱ و سپس **429** دقیقاً سر سقف.
- **e2e زنده (lockout):** ۵ تلاش ناموفق ⇒ ۴۰۱؛ سپس ورود با رمز درست با پیام
  «به‌دلیل تلاش‌های ناموفق، حساب موقتاً قفل شده است…» رد شد.

## ۵) پیکربندی
ثابت‌ها در کد: `MAX_FAILED_ATTEMPTS=5`، `LOCK_MINUTES=15` (auth.service)، و سقف‌های throttler
(app.module + auth.controller). برای استقرار چنداینستنسی، storage پیش‌فرضِ in-memory باید با
storage مبتنی بر Redis جایگزین شود (در H4/مقیاس‌پذیری realtime پوشش داده می‌شود).

## ۶) سازگاری عقب‌رو
کاملاً افزودنی است: ستون‌های جدید مقدار پیش‌فرض دارند؛ هیچ قرارداد API موجودی تغییر نکرد؛
رفتار ورود موفق بدون تغییر است. کاربران فعلی متأثر نمی‌شوند.
