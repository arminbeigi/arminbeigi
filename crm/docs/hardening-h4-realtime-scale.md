# سخت‌سازی H4 — مقیاس‌پذیری Realtime (آداپتور Redis برای Socket.IO)

> وضعیت: **اجراشده و اعتبارسنجی‌شده** (آزمون cross-instance زنده موفق). اولویت: 🟠 High.

## ۱) مسئله
Socket.IO با آداپتور حافظه‌ی پیش‌فرض فقط در یک اینستنس کار می‌کند: اگر دو نسخه‌ی API پشت
بارمتوازن‌کننده اجرا شوند، رویدادی که روی اینستنس A منتشر شود به کلاینت‌های متصل به اینستنس B
**نمی‌رسد** (پاپ‌آپ تماس از دست می‌رود). این مانع scale-out افقی بود.

## ۲) راهکار
- **`@socket.io/redis-adapter`** از طریق یک `RedisIoAdapter` سفارشی (زیرکلاس `IoAdapter`).
  هنگام بوت، اگر کلاینت Redis موجود باشد، آداپتور با کلاینت‌های pub و sub (با `duplicate()`)
  ساخته و روی سرور Socket.IO نصب می‌شود.
- **fallback خودکار**: اگر `REDIS_URL` تنظیم نشده باشد (کلاینت `null`)، به آداپتور حافظه‌ی
  پیش‌فرض برمی‌گردد (حالت تک‌اینستنس) — بدون شکست.
- از همان `RedisModule` سراسری (H3) استفاده می‌شود (یک منبع واحد برای Redis).

## ۳) فایل‌های افزوده/تغییر
- جدید: `src/common/adapters/redis-io.adapter.ts`.
- `src/main.ts` — استفاده از `RedisIoAdapter` و اتصال به Redis در صورت وجود.
- `package.json` — افزودن `@socket.io/redis-adapter`.

## ۴) شواهد اعتبارسنجی
- `npm run build` ⇒ بدون خطا.
- **e2e تک‌اینستنس** (رگرسیون): جریان realtime فاز ۷ ⇒ **۵/۵** سبز.
- **e2e cross-instance (اثبات اصلی):** دو اینستنس API (پورت‌های ۴۰۰۰ و ۴۰۰۱) روی همان
  Redis اجرا شدند؛ کلاینت به **اینستنس B** وصل شد، تماس روی **اینستنس A** شبیه‌سازی شد، و
  رویداد `call:incoming` از طریق **Redis pub/sub** به کلاینتِ B رسید ⇒ **۳/۳** سبز.

## ۵) سازگاری عقب‌رو
افزودنی و شفاف: در محیط تک‌اینستنس (یا بدون Redis) رفتار دقیقاً مثل قبل است. برای فعال‌سازی
scale-out کافی است چند نسخه‌ی API با همان `REDIS_URL` اجرا شوند (مثلاً `docker compose up
--scale api=3` پشت Nginx با sticky-session اختیاری).
